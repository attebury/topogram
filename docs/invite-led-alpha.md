# Invite-Led Alpha

This is the practical operating note for Topogram's invite-led alpha.

It answers three questions:

1. who is this for right now?
2. how should interested teams reach out?
3. what should they expect from the current alpha boundary?

## Who This Is For Right Now

Topogram is for technical teams already using coding agents who need stronger structure, review boundaries, and proof while evolving real software.

For the fuller fit profile, see [design-partner-profile.md](./design-partner-profile.md).

## Current Contact Path

The current alpha contact path is lightweight:

- open a GitHub issue in this repository
- use the `Alpha interest` issue template
- include:
  - the codebase or system shape
  - whether the main need is brownfield recovery, maintained-app evolution, or generated proof work
  - where agent-driven changes are currently drifting or hard to trust

This is intentionally simple for alpha. It is not yet a polished intake workflow.

## What Interested Teams Should Expect

The alpha is invite-led and scope-disciplined.

Expected strengths:

- brownfield recovery and reconcile/adopt workflows
- maintained-app proof and change-boundary guidance
- generated contracts, runtime bundles, and proof surfaces
- explicit safe, guarded, and no-go boundaries

Expected limits:

- auth is alpha-complete at the signed-token boundary, not production-ready
- deployment claims remain narrow
- example breadth is meaningful but not exhaustive
- some verification layers are still generated by the same system they validate

## Best Initial Trial Shape

The best first alpha engagement is usually one contained workflow:

- one brownfield repo or subsystem
- one workflow-heavy or contract-sensitive change
- one proof-oriented review loop with humans and agents both in the process

That is a better alpha fit than asking Topogram to be the full platform boundary for an entire product on day one.
